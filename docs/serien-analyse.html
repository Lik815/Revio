03

Kritische Lücken

⛓
FK-Verletzung: bookingRequestId zeigt ins Leere
Jeder via Inquiry erzeugte ScheduledSlot bekommt bookingRequestId: "inq_${id}". Diese ID existiert nicht in der BookingRequest-Tabelle. In SQLite (Dev) wird das still akzeptiert, weil SQLite FK-Enforcement standardmäßig aus ist. In PostgreSQL (Prod) wäre das ein Constraint-Fehler. Außerdem kann der Patient seine Inquiry-Termine niemals im Terminbereich sehen, weil die App Termine über BookingRequest lädt.
👁
Slot-Generator ignoriert Inquiry-Slots
generateAvailableSlots() prüft auf Overlap gegen BookingRequest WHERE status IN (PENDING, CONFIRMED). Via Inquiry bestätigte ScheduledSlots (Status SCHEDULED) werden nicht berücksichtigt. Folge: Patienten, die über das normale Buchungsformular buchen, können Slots belegen, die ein Inquiry-Patient bereits bestätigt hat — echtes Doppelbuchungsrisiko.
📅
Keine Seriengenerierung — 1 Slot statt N Slots
Der Confirm-Endpunkt nimmt frequenz und anzahlTermine aus dem Inquiry-Snapshot, nutzt sie aber nicht. Es entsteht immer genau 1 ScheduledSlot. Für eine Behandlungsserie (z.B. 6× KG, 1×/Woche) müssten 6 Slots auf aufeinanderfolgenden Wochentagen im entsprechenden Rhythmus erzeugt werden.
👤
Patient sieht bestätigte Inquiry-Termine nicht
TherapyTabPatient lädt Termine über myAppointments, das aus dem alten BookingRequest-System kommt. Inquiry-basierte ScheduledSlots haben keinen echten BookingRequest dahinter → der Patient sieht "Bestätigt" in der Anfragenliste, aber keinen Eintrag im Terminbereich.
🔄
Kein Konflikt-Handling für zukünftige Serie-Slots
Wenn die Serie generiert wird (z.B. 6 Wochen), können einzelne Termine auf bereits belegte Zeiten fallen (Urlaub, andere Buchung, Feiertag). Es gibt keine Logik, die Konflikte erkennt und entweder überspringt oder dem Therapeuten zur Auflösung zeigt.
04

Architektur-Entscheidung: Wie wird die Serie generiert?

Option A — Empfohlen

Automatisch beim Confirm

Therapeut bestätigt Starttermin + Rhythmus einmalig
System generiert N Slots automatisch
Passt zu frequenz/anzahlTermine im Schema
Patient sieht sofort alle kommenden Termine
Konflikte für spätere Slots möglich
Mehr Logik im confirm-Endpunkt
Option B

Manuell Termin für Termin

Therapeut hat volle Kontrolle
Keine Konflikt-Komplexität
Kein UX-Gewinn gegenüber Ist-Zustand
6 manuelle Bestätigungen pro Serie
Schema-Felder bleiben ungenutzt
Entscheidung: Option A. Der Therapeut gibt beim Confirm zusätzlich den Wochentag und die Uhrzeit für den Wiederholungsrhythmus an. Das System berechnet daraus die folgenden N−1 Termine und prüft jeden einzeln gegen den Slot-Generator. Konflikte werden dem Therapeuten angezeigt, er kann sie überspringen lassen.

Wie der Rhythmus aus frequenz abgeleitet wird

frequenzAbstand zwischen TerminenBeispiel ab Mo 04.08.
X17 Tage04.08 → 11.08 → 18.08 → …
X23–4 Tage (Mo+Do-Muster)04.08 → 07.08 → 11.08 → …
X32–3 Tage (Mo+Mi+Fr)04.08 → 06.08 → 08.08 → …

Für MVP: X2 = halbe Woche (3-4 Tage), X3 = alle 2 Tage. Exakter Abstand ist konfigurierbar ohne Schema-Änderung.

Migration-Strategie: bookingRequestId auf nullable setzen via prisma db push (SQLite Dev). Bestehende Slots haben weiterhin inq_${id} als bookingRequestId — das ist historisch falsch, aber harmlos in Dev. In einer Prod-Migration müssen diese auf inquiryId umgestellt werden.
06

API-Änderungen

Ändern POST /inquiry/:id/confirm
Neues Request-Schema: Zusätzlich zu datum, uhrzeitVon, uhrzeitBis kommt serieGenerieren: boolean (default true wenn suchtyp === 'SERIE').

Neue Logik für SERIE:
  1. Starttermin wie bisher validieren + Overlap-Check
  2. Aus frequenz und anzahlTermine die folgenden N−1 Termine berechnen
  3. Jeden Folgetermin gegen Slot-Generator prüfen → Konflikte markieren
  4. Transaktion: Inquiry CONFIRMED + N ScheduledSlots (je mit seriesIndex 1..N) + CapacityRule
  5. Response enthält slots: [{startsAt, endsAt, seriesIndex, conflict: boolean}]
Fix Slot-Generator: ScheduledSlots einbeziehen
generateAvailableSlots() in slot-generator.ts prüft aktuell nur BookingRequest WHERE status IN (PENDING, CONFIRMED) auf Overlap. Inquiry-basierte ScheduledSlots mit Status SCHEDULED werden ignoriert.

Fix: TherapistBlockedTime-ähnliche Overlap-Abfrage gegen ScheduledSlot WHERE status = 'SCHEDULED' hinzufügen, bevor Slots ausgegeben werden.
Neu GET /inquiry/:id/slots
Lädt alle ScheduledSlots einer Inquiry (für Therapeuten-Detailansicht und Patienten-Terminübersicht). Alternativ: in GET /inquiry/my und GET /inquiry/incoming die scheduledSlots-Relation einbinden.
Ändern POST /inquiry/:id/cancel
Aktuell: scheduledSlot.updateMany WHERE id = "slot_inq_${id}" → setzt genau einen Slot auf CANCELLED.
Neu: scheduledSlot.updateMany WHERE inquiryId = id AND status = 'SCHEDULED' → storniert alle offenen Serie-Slots.

Bekannte Risiken

⚡
Race Condition bei Serien-Confirm
Wenn zwei Patienten gleichzeitig denselben Therapeuten bestätigen, kann der Overlap-Check beide durchlassen. In SQLite (Dev) nicht kritisch. Für Prod: Advisory-Lock oder Serializable Transaction pro therapistId. Bereits in docs/dynamisches-buchungssystem.md §9 dokumentiert.
📆
Feiertage / Ausnahmen
Die Seriengenerierung beachtet keine Feiertage. Wenn ein berechnetes Datum auf einen Feiertag oder eine TherapistAbsence fällt, wird der Slot entweder übersprungen oder auf den nächsten verfügbaren Tag verschoben. MVP-Entscheidung: überspringen + dem Therapeuten melden, er bucht den Ersatztermin manuell nach.
🔢
anzahlTermine vs. Rezept-Kontingent
anzahlTermine kommt vom Patienten (default 6). GKV-Rezepte haben feste Einheiten (KG: 6–12 Einheiten). Für MVP: Wert wird ungeprüft übernommen. Follow-up: Validierung gegen Rezept oder PrescriptionData.
